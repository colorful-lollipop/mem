• 现状判断

现在这套代码其实已经有“收口方向”了，核心是 RpcClient 做统一 lifecycle owner，VesClient 只做策略装配和业务桥接，这一点在文档里写得很清楚，见 docs/architecture.md:90 和 docs/sa_integration.md:31。
但实现层还有几处会让人读着发虚，所以你感觉“生命周期、DFX、idle、crash、超时检测有点模糊”这个判断是对的。

我看下来，最主要的模糊来源有 4 个：

1. 文档状态机和代码状态机不完全一致。文档只列了 Uninitialized/Active/Cooldown/IdleClosed/Recovering/Closed，但代码里实际还有 Disconnected，见 memrpc/include/memrpc/client/rpc_client.h:135 和 docs/architecture.md:108。这会直接让人搞不清“恢复失败后到底是终态、临时态，还是可按需重连态”。
2. RpcClient 虽然已经集中做状态迁移，但“检测”和“执行恢复”还是揉在一起了。比如 watchdog 里同时做 pending timeout、health check、idle policy，然后直接触发 RequestExternalRecovery / HandleEngineDeath / ScheduleRecovery，见 memrpc/src/client/rpc_client.cpp:1380 和 memrpc/src/client/rpc_client.cpp:1426。读代码
   时很难一下看出“谁只负责发现问题，谁负责决策，谁负责状态迁移”。
3. VesClient 仍然保留了一层自己的恢复循环和派生语义。ShouldRetryRecoveryStatus、InvokeWithRecovery、lastObservedTrigger_、EngineDied() 本质上是在 RpcClient 统一状态机外面又包了一层“投影视图”，见 virus_executor_service/src/client/ves_client.cpp:24 和 virus_executor_service/src/client/ves_client.cpp:225。这
   不是错，但会让人误以为 VES 也拥有一套恢复状态机。
4. “健康检查协议语义”和“传输错误语义”混在一起。VesControlProxy::ReceiveSizedReply() 在 poll 超时、读失败时统一回 PeerDisconnected，heartbeat 再由上层翻译成 external recovery，见 virus_executor_service/src/transport/ves_control_proxy.cpp:194 和 virus_executor_service/src/transport/
   ves_control_proxy.cpp:336。对框架作者来说可以理解，但对后来维护的人来说，很容易分不清“真 crash / 心跳超时 / 协议坏包 / socket 断链”这几类事件的边界。

我建议的优化方向

第一优先级不是先改逻辑，而是先把“语义表”钉死，否则后面重构只会越改越散。

1. 先统一状态机字典。
   把 ClientLifecycleState、RecoveryTrigger、RecoveryAction 做成唯一术语表，并补齐 Disconnected 的正式定义。
   建议直接把状态分成三类：
   Live：Uninitialized/Active
   Recovering：Cooldown/Recovering
   Dormant：IdleClosed/Disconnected
   Terminal：Closed
   这样别人一眼就知道哪些状态还能自动恢复，哪些只能按需重连，哪些彻底终止。
2. 明确分层：检测、决策、执行、观测。
   当前目标文档里已经在往这个方向走了，但实现还可以再收一层。
   建议固定成四段：
   Detector：pending timeout、heartbeat、engine death、idle sampler
   Policy：把 detector 结果映射成 RecoveryDecision
   LifecycleExecutor：唯一负责 TransitionLifecycle / CloseLiveSession / OpenSession / FailAllPending
   Observer：RecoveryRuntimeSnapshot / RecoveryEventReport / health snapshot
   这样以后任何人找 bug，都知道应该去哪层看。
3. 让 VesClient 彻底变成“薄 facade”。
   VesClient 最好只保留：
   策略参数装配
   typed API 编解码
   AnyCall 降级
   DFX 回调桥接
   像 InvokeWithRecovery() 这种循环等待逻辑，如果保留，也应该明确标注它只是“调用层等待策略”，不是第二套 recovery state machine。否则读者很容易把 VesClient 当成恢复 owner。
4. 分开“故障原因”和“恢复动作”。
   现在代码里 trigger/action 已经分开了，这是对的；下一步建议在命名和日志里也严格分开：
   原因：ExecTimeout / EngineDeath / HealthTimeout / HealthMalformed / SessionMismatch / IdleThresholdReached
   动作：Restart / IdleClose / Ignore / ManualShutdown
   不要再用“recovery reason”这类宽词同时指代原因和动作。

更清晰易懂的落地顺序

如果你接下来要继续优化，我建议按这个顺序做，性价比最高：

1. 先改文档，不改行为。
   更新 docs/architecture.md:106，把 Disconnected、状态分类、合法转换、各状态能否自动恢复写全。
   这一项投入小，但能立刻降低团队沟通成本。
2. 再把 RpcClient 内部拆成几个显式 helper。
   现在已经有 TransitionLifecycle、ScheduleRecovery、EnterIdleClosed、EnterTerminalClosed，见 memrpc/src/client/rpc_client.cpp:652。
   下一步可以继续抽成：
   HandleDetectionEvent(...)
   DecideRecovery(...)
   ExecuteRecoveryDecision(...)
   PublishRecoveryObservation(...)
   这样 WatchdogLoop() 和 HandleEngineDeath() 会短很多。
3. 收敛 VesClient 的兼容视图。
   EngineDied() 最好明确改成“从最近一次 structured event 派生的测试/兼容 helper”，不要让它看起来像业务主语义。
   lastObservedTrigger_ 也建议改名成 lastNonReconnectTrigger_ 或直接删掉，用 RecoveryEventReport 现算。
4. 最后再细化健康检查错误模型。
   VesControlProxy 可以考虑把 heartbeat 失败更明确地区分成：
   Timeout
   TransportError
   ProtocolMismatch
   UnhealthyReply
   这样 RpcClient 收到 external signal 时更清楚，日志和 DFX 也更容易读。

一句话总结

你们现在不是“缺功能”，而是“语义已经在收口，但实现层还残留两层视图”:
RpcClient 是真实生命周期 owner，VesClient 和 VesControlProxy 还各自带着一点恢复/健康语义，所以读起来会糊。接下来的正确方向不是再加判断，而是继续做“单一状态机、单一 DFX 面、单一术语表”。

如果你要，我下一步可以直接帮你出一版“生命周期/DFX 重构说明文档”，或者直接给你做第一步文档收敛补丁。


› 可以， 还有我们的InvokeWithRecovery 好像也不应该在ves_client 里面， 应该收入到内部?


• 我先直接回答这个判断，然后把“放到哪一层更合适”说清楚，避免只是简单搬位置。

• 是，原则上 InvokeWithRecovery() 不应该长期留在 ves_client 里。

现在它在 virus_executor_service/src/client/ves_client.cpp:312 做的事情，本质不是 VES 业务语义，而是：

- 根据 RecoveryRuntimeSnapshot 判断当前是否处于 Cooldown/Recovering
- 在恢复窗口内等待
- 重新发起同一个调用
- 对 PeerDisconnected/CooldownActive 做统一重试

这已经是“客户端恢复调用编排”，不是 “VES ScanFile 业务逻辑”。
所以从分层上看，它更像应该收进 memrpc 内部，至少收进一个通用 client-side helper，而不是放在 VesClient facade 里。

为什么现在放在 VesClient 不理想

1. 它让 VesClient 看起来像第二个 lifecycle owner。
   实际 owner 是 RpcClient，但 InvokeWithRecovery() 又在外面做一层等待和重试，读代码的人会误判恢复语义分布。
2. 它和 VES 业务无关。
   这套逻辑对任何基于 RpcClient 的 facade 都可能成立，不是 ScanFile 专属能力。
3. 它会把 “调用等待策略” 和 “业务 fallback” 混在一起。
   尤其 VES 还有 AnyCall 降级路径，容易让人误会 memrpc lifecycle 和 AnyCall fallback 是一套逻辑。

更合理的落点

我建议分两步看：

1. 先收进 memrpc client 层，但不要直接塞进 RpcClient::InvokeAsync()。
   因为 InvokeAsync() 的语义现在很清楚，就是“一次提交一次 future”。如果直接把自动重试塞进去，会把 API 语义变重，也可能影响已有测试预期。
2. 在 memrpc 增一个明确的同步/包装层 helper，更合适。
   例如：

- RpcClient::InvokeWithRecovery(...)
- 或 RpcClient::InvokeWithRecovery(...)
- 或单独一个 ResilientInvoker / RecoveryAwareInvoker

这样语义更清楚：
InvokeAsync 是原子调用原语。
InvokeWithRecovery 是上层可选的恢复等待包装。

我更推荐的设计

把能力拆成两层：

- RpcClient
  负责 lifecycle、cooldown、reconnect、snapshot/report、单次调用原语
- RecoveryAwareInvoker 或 RpcClient::InvokeWithRecovery
  负责“当返回 CooldownActive/PeerDisconnected 且 snapshot 表示可恢复时，内部等待并重试”

这样 VesClient 只剩：

- 编解码
- 选择 memrpc 还是 AnyCall
- 调用通用 recovery-aware invoker

一个重要边界

不要把“自动重试整个业务调用”无脑下沉到 RpcClient 核心里。
因为一旦跨恢复窗口重试，就涉及 replay 语义，不是所有请求都一定安全。

你们现在 VesClient::InvokeWithRecovery() 实际重试的是“在请求还没真正成功发出去，或者 session 正在恢复时重新尝试进入调用”，这类语义还能接受。
但如果未来把它做成更通用能力，建议把边界写死：

- 只在 CooldownActive / PeerDisconnected 且明确处于恢复中的情况下重试
- 不对已经进入 pending 且可能被旧服务端看见的请求做透明重放
- 超大包 AnyCall 仍然只是 transport fallback，不参与 lifecycle owner 语义

结论

你的判断是对的：InvokeWithRecovery() 不该长期留在 ves_client。
但更准确地说，不是简单“搬进内部”，而是应该下沉为 memrpc client 层的通用恢复调用包装，而不是塞进 RpcClient 的最核心原语里。

1. memrpc 新增通用 InvokeSyncWithRecovery
2. VesClient 删除本地 InvokeWithRecovery
3. VesClient 只保留 codec + AnyCall fallback + policy wiring

这样改完结构会明显清楚很多。
